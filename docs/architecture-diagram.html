Hydra-Swarm v0.6.8.1

A local multi-agent development harness: a lead plans and judges, four vendor heads implement in isolated Git worktrees, and nothing a worker claims is real until it crosses a deterministic trust boundary.

A heterogeneous coding-agent control plane where untrusted probabilistic workers can move quickly — but only deterministic, replayable evidence can change trusted code.

The pipeline

Every arrow into or out of state passes through a harness launcher; the lead never hand-mutates authoritative state. Ledger + Git alone can reconstruct a run for a replacement lead.

The evolved thinking

Hydra-Swarm was not designed all at once. Each era below is a belief the project started with, the evidence that changed it, and the mechanism that belief became. The through-line never moved: deterministic gates decide; probabilistic tools inform.

Wave 0 · 2026-07-12 · the founding distrust

Agents report claims, not facts

Common practiceAn agent that says "tests pass, task complete" is done — read its summary and merge.
Hydra's moveEvery worker result is an untrusted claim. The harness re-derives the evidence itself: schema → Git evidence → ownership audit → sandboxed verify — before anything is promoted.

The founding decision was structural pessimism: treat coding agents like untrusted processes in a CI pipeline, not like junior colleagues. Workers get one bounded task, one branch, one worktree, an explicit ownership boundary — and no path to authoritative state. The very first hypothesis run (0002) validated the deepest version of this: two candidates that each passed verification alone failed the combined gate together, a planted semantic conflict a clean Git merge would have shipped. Parallel discovery, serialized convergence has been non-negotiable since.

Wave 1 · 2026-07-13 · judgment becomes contract

Review is a recorded protocol, not a vibe

Common practiceAsk a second model "does this look right?" and proceed on a thumbs-up.
Hydra's moveCross-vendor review with recorded verdicts under a tracked policy — a different model family reviews each candidate, only accept proceeds, and the verdict is a ledger event.

Wave 1 turned the lead's ad-hoc judgment into contracts: risk-triggered review policy, capability profiles fed by measured usage evidence rather than vendor marketing, and the rule that instructions are ledger events — no mid-turn whispering to a running worker; course-correction is a version-bumped spec amendment. The same era set the recovery bar that still shapes everything: a replacement lead must reconstruct any run from Git plus the state store alone. Conversational context is never load-bearing.

Wave 2 · 2026-07-13 · heterogeneity as strength

Confinement matched to trust, not one-size-fits-all

Common practiceOne uniform sandbox (or none) for every agent.
Hydra's moveVendor-asymmetric confinement: Kimi auto-approves its own tools, so its write role is refused without a mandatory srt OS sandbox with a network allowlist derived from the worktree's own dependency manifests. Codex brings its own workspace-write sandbox. Claude is bounded by structure + the post-hoc audit — a gap the docs state honestly and the daemon milestone closes.

All four heads arrived, and with them the insight that different vendors deserve different boundaries. The same era added GitNexus and Graphify under a strict constitution: code intelligence is advisory, never a gate — extracted graph edges open investigations, inferred edges only pose review questions. Nothing probabilistic acquired veto or approval power.

Runs 0040–0046 · the runtime hardening

Fail loudly; degrade never-silently

Common practiceKeep legacy code paths alive as silent fallbacks "just in case."
Hydra's moveThe Bash lane was retired loudly — HYDRA_HARNESS=bash errors instead of coercing. An explicit HYDRA_HARNESS=bin never silently downgrades. Implicit defaults may fall back; explicit operator choices may not.

The harness itself went from Bash to TypeScript to a Bun-compiled single binary as the default runtime — the only lane that works with no Node on PATH. The distinction that emerged is now a house principle: a default can degrade gracefully (fresh checkout, no binary → source lane, silently fine), but an explicit operator request must hard-fail rather than quietly substitute. The same principle shows up in dispatch: a missing assigned_vendor fails fast naming available substitutes — the harness never auto-substitutes a head; humans re-pin.

Runs 0047–0050 · v0.6.4–0.6.6 · the environment is part of the contract

Detect, preflight, verify — before the worker exists

Common practiceSpawn the agent and let it discover mid-task that a tool is missing.
Hydra's movedetect-heads probes what is actually installed (down to opencode's configured model list); worker-devenv preflight verifies network domains, caches, and the toolchain before spawn; HYDRA_NODE_BIN pins a verified Node so a login-shell PATH rebuild can't strand a worker on a stale system node.

A string of field incidents (a v17 system node shadowing v22 inside a vendor tool shell; srt's physical-path matching silently defeating a pnpm-store fix because macOS TMPDIR lives behind a /var symlink) taught that the worker's environment is as much a correctness surface as its diff. Binary distribution absorbed the same doctrine: a release binary installs only if the manifest SHA-256, its self-reported version, and its target triple all match — a stale binary is structurally invisible, not merely warned about.

Run 0048 · v0.6.7 · lifecycle closure

Document, then delete

Common practiceWorktrees accumulate forever, or get bulk-deleted on faith.
Hydra's movehydra run-log renders a complete per-run audit document first; hydra gc then reaps only worktrees the ledger proves integrated — recorded SHA reachable from the default branch, proof paired to the current tip, revalidated before each destructive op, compare-and-delete atomic.

Deletion became the last state transition of a documented lifecycle instead of an act of housekeeping faith. Notably, gc is designed to refuse what it cannot prove: PR-squash-merged history is unprovable from the ledger by design, so those worktrees take a manual, human-attested lane. Eleven worker attempts and seven adversarial review rounds closed eighteen deletion-safety findings before this shipped — the pipeline applied to itself.

Runs 0051–0055 · v0.6.8.1 · observability as obligation

No blank panes, no silent hangs

Common practiceIf the agent produced no visible output, wait and hope.
Hydra's moveEvery vendor × every dispatch mode must show live progress or an explicit heartbeat — a blank pane is a bug (found live in run 0050, fixed as #18). A provider outage that once hung dispatches for a full 50-minute timeout now trips a usage_limited terminal state plus a machine-global cooldown that refuses to start the next doomed dispatch.

The loop-thinking detector embodies the same philosophy from earlier eras: it measures real Git progress, not output-text repetition — the only signal that actually matters — and auto-cancels through the same clean path an operator would use. Even the terminal layout answers to the doctrine: agent panes shrink to a configured ratio so the lead's console — where judgment happens — keeps the majority of the screen.

2026-07-19 · the discipline era

Version numbers are claims too

Common practiceShip fast, bump fast — 0.7, 0.8 within days.
Hydra's moveThe published 0.7.x/0.8.x series was deliberately renumbered back to 0.6.4–0.6.8: the pace overstated the changes. Features bump the third segment; fixes bump a fourth (0.6.8.1). No minor bump without operator sign-off.

If the whole system is built on honest claims, the version string doesn't get an exemption. The same week, a cross-vendor deep-research consultation (run 0050 — Codex and OpenCode researching independently, consolidated by the lead) confirmed the market position: checkpointing, schedulers, and web dashboards are commodities; a deterministic, code-specific promotion boundary across heterogeneous vendors is not. The prioritized next steps sharpen that moat: an adversarial verification vault (verifier-owned held-out evidence workers cannot read — the answer to documented reward-hacking) and a PR-mode trust check ("we don't review your PR — we re-verify it"). Explicitly rejected as hype: swarm consensus topologies, opaque self-learning memory, autonomous merge on agent-authored tests.

Current capability notes

Head detection (v0.6.4)

  • detect-heads probes all four vendor CLIs, enumerates opencode's configured models and active model, and checks srt for Kimi write-capability.
  • Snapshot: ~/.local/state/hydra/heads.json, refreshed by run-init (ledger event heads_detected).
  • allocate never recommends an uninstalled head; dispatch fails fast naming available substitutes — never auto-substitutes.

Runtime & distribution

  • Default: Bun-compiled single binary. fetch-bin.sh downloads the release matching this plugin's version, gated on manifest SHA-256 + self-reported version + target triple; npm run build:bin builds locally; source lane falls back automatically.
  • hydra version reports the version the binary was built from; hydra help lists every subcommand with its arguments.

Run close (v0.6.7)

  • hydra run-log <run> renders the per-run audit document to docs/hydra-dev-logs/ (document, then delete).
  • hydra gc --apply --keep-last 3 reaps only ledger-proven-integrated worktrees; dry-run by default; every destructive op revalidated and compare-and-deleted atomically.

Live feedback everywhere (v0.6.8.1)

  • Review-lane live tails cover all vendors; Claude shows a heartbeat when it can't stream — no dispatch mode leaves a blank pane.
  • Agent panes shrink via HYDRA_HERDR_PANE_RATIO (default 0.25); panes stay pinned to the run's originating workspace.
  • Real Kimi session resume (-S); amended tasks can carry a machine-checkable amendment_check gate; vendor usage-limit detection + pre-dispatch cooldown.
