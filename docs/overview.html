local multi-agent dev harness

HYDRA-SWARM

One lead plans and judges. Four vendors — Claude, Codex, OpenCode, Kimi — implement in isolated worktrees. Nothing reaches your tree without crossing the trust boundary.

v0.6.8.1 MIT licensed compiled binary default gc + run-log lifecycle
the idea

Agents propose. The harness decides.

Hydra-Swarm treats every worker — human-directed or autonomous — as untrusted until it clears a deterministic gate. The lead never hand-edits authoritative state; workers never touch it at all. Every promoted candidate can be reconstructed from Git and an append-only ledger alone, even if the lead session that started it is gone.

the trust boundary

Six stages, one gate

A candidate moves through the same pipeline whether it came from Claude, Codex, OpenCode, or Kimi. Nothing skips a stage.

STAGE 01

Plan

The lead writes one task spec per lane — objective, acceptance criteria, a disjoint writable_paths boundary. No two writers share a tree.

run-init.sh · create-worktree.sh
STAGE 02

Dispatch

The task routes to its assigned vendor, hosted in an isolated worktree with its own branch. Live progress streams to a monitor pane where the vendor supports it.

dispatch.sh <run> <task>
STAGE 03

Promote

The actual trust boundary: schema check → Git evidence → ownership audit → sandboxed verify. The raw inbox drop is an untrusted claim until it clears every gate.

promote.sh
STAGE 04

Review

Cross-vendor by convention — Codex reviews Kimi's candidate, OpenCode reviews Codex's fix. Only accept lets a candidate proceed.

review-dispatch.sh
STAGE 05

Squash

The harness — never the worker — squashes an accepted candidate into one clean commit.

squash.sh
STAGE 06

Integrate

Dependency-ordered cherry-picks, a per-candidate smoke verify, then a combined gate. Merge, push, and deploy stay human-authorized.

integrate.sh
the roster

Four vendors, one contract

Every adapter speaks the same argument / stdout / exit-code contract, so the pipeline above never needs to know which one it's talking to.

Claude

The lead's own hand

Runs non-streaming — a single JSON response at exit, no live capture. Loop detection degrades gracefully rather than guessing from silence.

adapter-claude.ts
Codex

The adversary of choice

Streams a real JSONL event log. Dispatched to review nearly every high-risk candidate this project has shipped — three rounds, for the loop detector alone.

adapter-codex.ts
OpenCode / GLM

The outside eye

Hosted as a decoupled monitor process rather than inside a pane — direct pane-hosting broke its CLI outright. Caught a wrong-process-signal bug nobody else found.

adapter-opencode.ts
Kimi

The default implementer

Stdout NDJSON tailed live into the pane. The usual first hand on a new task — implement, get reviewed, fix, repeat until accepted.

adapter-kimi.ts
hang detection · task #31

An agent that's alive isn't the same as an agent that's working

A worker can stream output forever without producing anything — the same failing command, the same three-step cycle, on repeat. Hydra-Swarm now has an answer to that on both ends: a way to check in, and a way to make sure something is actually watching.

01 · READ-ONLY

status.sh

One-shot check on a running task: ledger state, elapsed time against the timeout and hard-cap budgets, advisory process liveness, a progress tail, the last five ledger events. No pane-tailing required.

02 · SAFE CANCEL

cancel-task.sh

The only supported way to stop a dispatch cleanly. Resolves the real dispatch process — never the worker directly — sends SIGTERM, waits for its own terminal ledger write, escalates to SIGKILL only as a last resort with identity re-checked immediately before signaling.

03 · AUTONOMOUS

Loop-thinking detector

Watches Git worktree activity against repeated action signatures. Two stages — suspected, then confirmed — before it cancels through the exact same clean path as a human-issued cancel. Off by default for no vendor; opt out per task with HYDRA_LOOP_DETECTOR=0.

what "trust boundary" actually cost

Nothing that cancels work on its own ships on the first pass

The loop detector can autonomously kill a running task, so it was held to a higher bar than everything else in this release — three real adversarial rounds before merge.

Round 1 — Rejected

Four high-severity false-positive paths: stale evidence surviving real progress, a confirmation stage that didn't require continued evidence, a rule that could fire on repeated successful work, Git sampling that wasn't fully fail-open.

Round 2 — Partially fixed

Four of eight findings closed outright. The rest narrowed but not closed: one stale failure could still hide behind a run of successes and reach auto-cancel.

Round 3 — Accepted

Every residual closed and independently re-traced against the exact counterexamples from round one. No new issue introduced by the fix itself.
